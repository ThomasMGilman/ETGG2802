void reap(vector<Boss> bosses ){
    unsigned i=0;

        if( bosses[i].x > 100 )
            bosses.erase( bosses.start()+i );

Problem
  • This makes a copy of bosses, reaps the data from the copy, and then discards the copy!
  • We need to pass pointer to bosses list:
void reap(vector<Boss>* bosses ){
    unsigned i=0;
    while( i < bosses->size() ){
        if( (*bosses)[i].x > 100 )
            bosses->erase( bosses->start()+i );
        else
            i += 1
    }
}

//elsewhere
std::vector<Boss> bosses;
bosses.push_back( Boss( 1,2 ) )
...
reap(&bosses);
Note
  • The second C++ example has the same result as the Python example
  • Python has pointer semantics but not pointer syntax.

Generics
  • Suppose we want to write a reap() function.

def reap(L):
    i=0

Memory Leak
  • C++ does not automatically free memory when it becomes unreachable
  • Memory was allocated when we put something in the list:
//declaration
vector<Bullet*> bullets;

//put something in the list
bullets.push_back( new Bullet(x,y) );
  • When this bullet is removed from the array, the memory is still allocated
Solution
  • We need to deallocate the memory:
void reap( vector<Entity*>& L ){
    unsigned i=0;
    while( i < L.size() ){
        if( L[i]->isDead() ){
            delete L[i];
            Entity* x = L.back();
            L.pop_back();
            if(i < L.size() )
                L[i] = x;
        } else {
            i+=1;
        }
    }
}
Problem
  • It‚Äôs easy to forget to do the deallocation
  • We might have lots of code that manipulates collections
  • Lots of potential for bugs
  • Better way: Have the computer do the work for us
Solution
  • We can use smart pointers
  • Declare the vector like so:
vector< std::shared_ptr<Bullet> > bullets;
  • Put item in the list:
bullets.push_back( std::make_shared<Bullet>(x,y) );
  • Now memory is automatically deallocated
Question
  • Why not use shared_ptr‚Äôs everywhere by default?
    • A little slower than ordinary pointers
    • If we make mistake in code (syntax error): Compiler diagnostics are harder to understand
    • shared_ptr‚Äôs don‚Äôt work if we have circular references
Option 2
  • There‚Äôs another way: We can create a template function

Problem
  • What if we want to take action when a specific thing is removed
    • Ex: Whenever a Bullet is reaped, want to make a ParticleSystem
  • This is a common problem, so a solution pattern has been developed: Callback function
    • Whenever reap() removes an object, it will call a function that we provide
Code
template<typename T, typename F>
void reap(std::vector<T>& L, const F& callback )
{
    unsigned i=0;
    while(i<L.size()){
        if(L[i].isDead()){
            callback(L[i]);
            T x = L.back();
            L.pop_back();
            if( i < L.size() )
                L[i] = x;
        } else {
            i+=1;
        }
    }
}

Analysis
  • Templates can be very useful
    • Allow us to emulate some weakly typed language features
  • But: They also partially circumvent C++‚Äôs type checks
    • Can lead to hard-to-find-bugs
  • And: Compiler error messages are sometimes hard to understand
