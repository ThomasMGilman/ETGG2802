Noise
Motivation
  • Want to add imperfections (noise) to our rendering
    • Noise = Visual noise, not audio noise
  • But why?
Motivation
  • Real world isn’t perfect
    • Dust, dirt, scratches
    • Chaotic natural processes
      • Fire, smoke, clouds
    • Irregular patterns
      • Camouflage
Noise
  • Noise: “Random” values
    • But…as usual, it’s not quite that simple!
  • We want to create some functions:
    • float noise( float v ){ … }
    • float noise( vec2 v ){ … }
    • float noise( vec3 v ){ … }
    • float noise( vec4 v ){ … }
  • Input: Some scalar or vector value
  • Output: Floating point value in range [-1, 1]
Noise
  • Important restrictions:
    • Same input value should always give same output
    • Returned value should “look” random
Option 1
  • Suppose we use v as input to a random number generator
  • How can we convert a value to a random number?
PRNG
  • Usually, when we generate random numbers on a computer, we use a pseudorandom number generator (PRNG)
  • Virtually all PRNG’s maintain internal state that is used to generate random numbers
Xorshift
  • One choice: The Xorshift generator:
float randomNumber(inout uint state){
    state ^= (state<<13);
    state ^= (state>>17);
    state ^= (state<<15);
    return (state & 0x7fffffff)/float(0x7fffffff);
}
LCG
  • Another choice: A linear congruential generator
  • On my system, this was about 8% faster than Xorshift
    • But: The RNG’s statistical properties probably aren’t as good as Xorshift (but it shouldn’t matter for our purposes here)
    • And: This didn’t work on my Intel chipset… :-/
float randomNumber(inout uint state){
    state = 16807*state;
    return (state & 0x7fffffff) / float(0x7fffffff);
}
Example
  • Easy noise shader:
void main(){
    vec2 v = gl_FragCoord.xy;
    uint x = uint(v.x);
    float v1 = randomNumber(x);
    uint y = uint(v.y);
    float v2 = randomNumber(y);
    float f = (x^y)/float(0x7fffffff);
    color = vec4(f, f, f, 1.0);
}
Results
  • Not very random!
  • Why not?
    • PRNG’s are good at producing uncorrelated sequences
    • But if there’s structure to the seeds, results are very un-random!
Solution
  • We can use a hash function to improve things
  • One useful function: A Wang hash
uint wangHash(uint v){
    v = 9*( (v >> 16 ) ^ (v ^ 61));
    v ^= (v>>4);
    v = v * 668265261;
    return v ^ (v>>15);
}
Code
  • Now we use it like so:
void main(){
    //vec2 v = v_texcoord * 100;
    vec2 v = gl_FragCoord.xy;
    uint x = uint(v.x);
    x = wangHash(x);
    float v1 = randomNumber(x);
    uint y = uint(v.y);
    y = wangHash(y);
    float v2 = randomNumber(y);
    float f = (x^y)/float(0x7fffffff);
    color = vec4(f, f, f, 1.0);
}
Result
  • Better…But there’s still an odd artifact (diagonal line)
  • And the noise is too random!
    • Values change quickly from pixel to pixel
    • Tends to produce visible artifacts
  • We need something a little less random!
Coherent Noise
  • Coherent noise: Much better
    • Varies smoothly from one point to another
    • Developed by A. Perlin
    • Used in many Hollywood movies (CG)
    • Academy award winner (!)
Perlin Noise
  • Over the years, Perlin has refined his noise algorithm
    • Original
    • Improved
    • Simplex
  • The original one is the easiest to understand, so that’s what we’ll look at here
  • Initially, we’ll look at 2D noise
    • Input: vec2
    • Output: float between -1…1
Integer Lattice
  • Let q0, q1, q2, q3 be coordinates of 4 closest integer lattice points around p
  • Ex: If p = (3.8, 12.2):
    • q0 = (3, 13)
    • q1 = (4, 13)
    • q2 = (3, 12)
    • q3 = (4, 12)
Integer Lattice
  • We define a random unit length vector at every integer lattice point
    • These are called gradient vectors
  • Let g0, g1, g2, g3 be the gradient vectors at q0, q1, q2, q3 respectively
  • We’ll see how we get the g’s in a moment
Vectors
  • Compute four vectors:
    • v0=p-q0
    • v1=p-q1
    • v2=p-q2
    • v3=p-q3
Dot Products
  • Compute four dot products:
    • d0=g0 ⋅ v0
    • d1=g1 ⋅ v1
    • d2=g2 ⋅ v2
    • d3=g3 ⋅ v3
Rationale
  • Dot products vary smoothly depending on p’s position
    • As p moves within grid cell, the four dot products will smoothly increase or decrease
    • Note: If p is right on one of the integer lattice points, the associated dot product is zero
Interpolation
  • Next, we perform bilinear interpolation:
  • First, interpolate between d0 and d1 and then between d2 and d3:
  • t0 = (px − q0x)/(q1x − q0x)
    • In GLSL: t0 = fract(p.x)
  • a0=d0+t0(d1-d0)
    • In GLSL: a0 = mix( d0, d1, t0 );
  • a1=d2+t0(d3-d2)
    • In GLSL: a1 = mix( d2, d3, t0 );
Interpolation
  • Then do another round of interpolation:
    • t1 = (py − q2y)/(q0y − q2y)
      • GLSL: t1 = fract(p.y);
    • n = a1 + t1(a0 − a1)
      • GLSL: n = mix(a1, a0, t1 );
  • n is the noise value we use
Results
  • This almost works
  • Output looks too “blocky”
Fix
  • A simple fix: When we compute t0 and t1 do so like this:
  • t0 = 3⎛⎝(px − q0x)/(q1x − q0x)⎞⎠2 − 2⎛⎝(px − q0x)/(q1x − q0x)⎞⎠3
    • In GLSL: t0 = smoothstep(0.0, 1.0, fract(p.x));
  • Likewise for t1
  • This gives better (smoother) results
Note
  • Sometimes, we use the noise value to perturb surface normals
  • Problem: smoothstep() is not C2 continuous
  • We can compute t0 = fade(fract(px)) and t1 = fade(fract(py))
    • Where fade(x) → x*x*x*(10+x*(-15+6*x))
  • This is C2 continuous and avoids artifacts
Gradients
  • There’s just one little detail we left out…How to get the gradient vectors
  • Two possible approaches: Table based vs pure computation based
  • On my computer, table-based was a bit faster, but computation-based involves less setup
  • So we’ll look at that approach
    • Table-based is discussed in the appendix
Gradient
  • We want a function: vec2 gradient(vec2 v1)
    • Input: A 2D point in space
    • Output: A random 2D unit vector
Gradient
vec2 gradient(vec2 v1){
    ivec2 v=ivec2(v1);  //convert to int
    v &= 0xffff;          //keep low 16 bits
    uint state= wangHash( (v.y << 16) | v.x ); //hash
    float vx = randomNumber(state);  //get random x
    float vy = randomNumber(state);  //get random y
    vec2 r = vec2(vx, vy);   //build vector
    r -= vec2(0.5);     //put in range -0.5 ... 0.5
    r = normalize(r);   //make unit length
    return r;
}
Noise
  • OK, so what does the final noise function look like?
float noise(vec2 p){
    float x = p.x, y=p.y;
    vec2 q[4];
    q[0] = vec2(floor(x), ceil(y));
    q[1] = vec2(ceil(x), ceil(y));
    q[2] = vec2(floor(x), floor(y));
    q[3] = vec2(ceil(x), floor(y));
    float d[4];
    for(int i=0;i<4;++i){
        vec2 g = gradient(q[i]);
        vec2 v = p-q[i];
        d[i] = dot(g, v);
    }
    float t = fract(p.x);
    t = smoothstep(0.0, 1.0, t);
    float a[2];
    a[0] =  mix( d[0], d[1], t );
    a[1] =  mix( d[2], d[3], t );
    t = fract(p.y);
    t = smoothstep(0.0, 1.0, t);
    float n = mix(a[1], a[0], t );
    return n;
}
Repetition
  • Note that our noise repeats every 65535 units in x and/or y
  • Usually, that’s not a problem
Other Dimensions
  • Sometimes, we want 1D or 3D noise
  • This is a straightforward extension of previous code
Gradient
  • We need 1D gradient function
  • A 1D vector is just a scalar! (It will be either -1 or 1)
float gradient(float v1){
    int v = int(v1);
    v &= 0xfffff;
    uint state= wangHash( v );
    float vx = randomNumber(state); //vx in 0...1
    vx = float(vx<0.5);     //0 or 1
    vx = vx - 0.5;      //-0.5 or 0.5
    return vx*2.0;      //-1 or 1
}
  • Repeats every 1,048,576 units
Noise
  • And then we compute the 1D noise value:
float noise(float p){
    float q[2];
    q[0] = floor(p);
    q[1] = ceil(p);
    float d[2];
    for(int i=0;i<2;++i){
        float g = gradient(q[i]);
        float v = p-q[i];
        d[i] = dot(g, v);
    }
    float t = fract(p);
    t = smoothstep(0.0, 1.0, t);
    float n =  mix( d[0], d[1], t );
    return n;
}
  • Notice we have half as many computations as the 2D case
3D
  • What about 3D noise?
  • We must create a 3D gradient function:
vec3 gradient(vec3 v1){
    ivec3 v=ivec3(v1);
    v &= 0xfff;
    uint state= wangHash( (v.z << 24) | (v.y << 12) | v.x );
    float vx = randomNumber(state);
    float vy = randomNumber(state);
    float vz = randomNumber(state);
    vec3 r = vec3(vx, vy, vz);
    r -= vec3(0.5);
    r = normalize(r);
    return r;
}
  • This one repeats every 4096 units in x, y, and/or z
3D
  • Finally, our 3D noise function. Just a straightforward extension of the 2D case into 3D
    • But twice as many computations!
float noise(vec3 p){
    float x = p.x; float y = p.y; float z = p.z; float t;
    vec3 q[8];  float d[8];  float a[4]; float b[2];
    q[0] = vec3(floor(x), ceil(y), floor(z));   q[1] = vec3(ceil(x), ceil(y), floor(z));
    q[2] = vec3(floor(x), floor(y), floor(z));  q[3] = vec3(ceil(x), floor(y), floor(z));
    q[4] = vec3(floor(x), ceil(y), ceil(z));    q[5] = vec3(ceil(x), ceil(y), ceil(z));
    q[6] = vec3(floor(x), floor(y), ceil(z));   q[7] = vec3(ceil(x), floor(y), ceil(z));
    for(int i=0;i<8;++i){
        vec3 g = gradient(q[i]);
        d[i] = dot(g, p-q[i]);
    }
    t = smoothstep(0.0, 1.0, fract(p.x));
    a[0] =  mix( d[0], d[1], t );    a[1] =  mix( d[2], d[3], t );
    a[2] =  mix( d[4], d[5], t );    a[3] =  mix( d[6], d[7], t );
    t = smoothstep(0.0, 1.0, fract(p.y));
    b[0] = mix( a[1], a[0], t );    b[1] = mix( a[3], a[2], t );
    t = smoothstep(0.0, 1.0, fract(p.z) );
    return mix(b[0], b[1], t );
}
4D
  • We can even produce 4D noise…But this is pretty computationally heavy
  • Input: vec4 p = (x, y, z, w)
    • 4th dimension = succession of cubes (time dimension)
    • Do 3D interpolation in two cubes using x, y, z in each cube
      • w gives which cube to use
    • This gives two averages: avg1, avg2
    • Compute weighted average:
      • t=fpart(p.w)
      • pct=3*t*t-2*t*t*t
      • avg=avg1+t*(avg2-avg1)
  • Can also do 5D noise: p=(x, y, z, w, u)
    • Compute two 4D noise values and interpolate between with u coordinate
  • Time required: O(2k) where k=num dimensions

Appendix
  • Noise with table lookup
  • Useful for mobiles or other platforms without fast GLSL integer instructions
Permutation
  • On CPU: Compute permutation table:
std::uint8_t P[256];
for(int i=0;i<256;++i)
    P[i] = i;
std::shuffle(P, P+256, std::minstd_rand());
Gradient
  • Compute gradient table
    • vec4 G[256]
    • Fill G with 256 random unit-length vectors
Textures
  • Create two textures with the data from P and G
DataTexture2DArray* Ptex = new DataTexture2DArray(256, 1, 1, GL_R8UI, GL_RED_INTEGER, GL_UNSIGNED_BYTE, P );
DataTexture2DArray* Gtex = new DataTexture2DArray(256, 1, 1, GL_RGBA32F, GL_RGBA, GL_FLOAT, G );
Textures
  • Bind them:
//Create a sampler with filter mode set to nearest
//and bind it to 8 & 9
globs->nearestsampler.bind(8);
globs->nearestsampler.bind(9);
Ptex->bind(8);
Gtex->bind(9);
GLSL
  • Shader functions:
layout(binding=8) uniform usampler2DArray Ptex;
layout(binding=9) uniform sampler2DArray Gtex;

int P(int idx){
    return int(texelFetch( Ptex, ivec3( (idx) & 255 , 0 , 0), 0).r);
}
vec4 G(int idx){
    return texelFetch( Gtex, ivec3( (idx) & 255 , 0, 0 ), 0);
}
float gradient(float vf){
    int v = int(vf);
    return G( P(v) ).x;
}
vec2 gradient(vec2 vf){
    ivec2 v = ivec2(vf);
    return G( P(v.x + P(v.y)) ).xy;
}
vec3 gradient(vec3 vf){
    ivec3 v = ivec3(vf);
    return G( P(v.x + P( v.y + P(v.z)))).xyz;
}
vec4 gradient(vec4 vf){
    ivec4 v = ivec4(vf);
    return G( P(v.x+ P( v.y+ P( v.z + P(v.w)))));
}
  • If we need to target older GPU’s that can’t do bitwise operations in shader:
    • AND with 255 == MOD with 256
Gradients
  • Net effect: Random vectors for x, y, z in 0…255 and then it repeats
    • Repetition usually not noticeable
    • We don’t often render noise directly
    • Could use larger table (512, 1024, etc.)
      • Change array size + the bitwise-and operation
Precomputation
  • If we’re only using 1D or 2D noise: Could compute entire noise texture once on CPU and then upload to GPU
    • Takes more RAM, but might be faster
    • Infeasible for 3D noise (too much memory)
    • Not possible for 4D noise (no 4D textures in GL)
